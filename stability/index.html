Stability

If we solve a problem using a computer algorithm and see a large error in the result, we might suspect poor conditioning in the original mathematical problem. But algorithms can also be sources of errors. When error in the result of an algorithm exceeds what conditioning can explain, we call the algorithm unstable.

1.4.1Case study¶

In Example 1.2.4 we showed that finding the roots of a quadratic polynomial ax2+bx+cax^2 + b x+cax2+bx+c is poorly conditioned if and only if the roots are close to each other relative to their size. Hence, for the polynomial

p(x)=(x−106)(x−10−6)=x2−(106+10−6)x+1,p(x) = (x-10^6)(x-10^{-6}) = x^2 - (10^6+10^{-6})x + 1,p(x)=(x−106)(x−10−6)=x2−(106+10−6)x+1,
(1.4.1)

finding roots is a well-conditioned problem. An obvious algorithm for finding those roots is to directly apply the familiar quadratic formula,

x1=−b+b2−4ac2a,x2=−b−b2−4ac2a.x_1 = \frac{-b + \sqrt{b^2-4ac}}{2a}, \qquad x_2 = \frac{-b - \sqrt{b^2-4ac}}{2a}.x1​=2a−b+b2−4ac​​,x2​=2a−b−b2−4ac​​.
(1.4.2)

Example 1.4.1 suggests that the venerable quadratic formula is an unstable means of computing roots in finite precision. The roots themselves were not sensitive to the data or arithmetic—it’s the specific computational path we chose that caused the huge growth in errors.

We can confirm this conclusion by finding a different path that avoids subtractive cancellation. A little algebra using (1.4.2) confirms the additional formula x1x2=c/ax_1x_2=c/ax1​x2​=c/a. So given one root rrr, we compute the other root using c/arc/arc/ar, which has only multiplication and division and therefore creates no numerical trouble.

The algorithms in Example 1.4.1 and Example 1.4.2 are equivalent when using real numbers and exact arithmetic. When results are perturbed by machine representation at each step, though, the effects may depend dramatically on the specific sequence of operations, thanks to the chain rule (1.2.9).

Observation 1.4.1

The sensitivity of a problem f(x)f(x)f(x) is governed only by κf\kappa_fκf​, but the sensitivity of an algorithm depends on the condition numbers of all of its individual steps.

This situation may seem hopelessly complicated. But the elementary operations we take for granted, such as those in Table 1.2.1, are well-conditioned in most circumstances. Exceptions usually occur when ∣f(x)∣|f(x)|∣f(x)∣ is much smaller than ∣x∣|x|∣x∣, although not every such case signifies trouble. The most common culprit is simple subtractive cancellation.

A practical characterization of instability is that results are much less accurate than the conditioning of the problem can explain. Typically, one should apply an algorithm to test problems whose answers are well-known, or for which other programs are known to work well, in order to spot possible instabilities. In the rest of this book we will see some specific ways in which instability is manifested for different types of problems.

1.4.2Backward error¶

In the presence of poor conditioning for a problem f(x)f(x)f(x), even just the act of rounding the data to floating point may introduce a large change in the result. It’s not realistic, then, to expect any algorithm f~\tilde{f}f~​ to have a small error in the sense f~(x)≈f(x)\tilde{f}(x)\approx f(x)f~​(x)≈f(x). There is another way to characterize the error, though, that can be a useful alternative measurement. Instead of asking, “Did you get nearly the right answer?”, we ask, “Did you answer nearly the right question?”

Definition 1.4.1 (Backward error)

Let f~\tilde{f}f~​ be an algorithm for the problem fff. Let y=f(x)y=f(x)y=f(x) be an exact result and y~=f~(x)\tilde{y}=\tilde{f}(x)y~​=f~​(x) be its approximation by the algorithm. If there is a value x~\tilde{x}x~ such that f(x~)=y~f(\tilde{x}) = \tilde{y}f(x~)=y~​, then the relative backward error in y~\tilde{y}y~​ is

∣x~−x∣∣x∣.\frac{ |\tilde{x}-x| } { |x| }.∣x∣∣x~−x∣​.
(1.4.3)

The absolute backward error is ∣x~−x∣|\tilde{x}-x|∣x~−x∣.

Backward error measures the change to the original data that reproduces the result that was found by the algorithm. The situation is illustrated in Figure 1.4.1.

Small backward error is the best we can hope for in a poorly conditioned problem. Without getting into the formal details, know that if an algorithm always produces small backward errors, then it is stable. But the converse is not always true: some stable algorithms may produce a large backward error.

Example 1.4.4
